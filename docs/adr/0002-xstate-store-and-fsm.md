# State is split: `@xstate/store` for the chain, `xstate` FSM for the UI flow

Two packages, deliberately. `@xstate/store` holds the **edit chain** as data (`Layer[]`) with event-based mutations (`add` / `remove` / `reorder` / `updateParams`) and is persist-ready. `xstate` holds the **UI flow** (panel mode + `selectedLayerId`) as a finite state machine so the invariant `mode === "edit" ⇒ selectedLayerId !== null"` is enforced by construction, not by convention. The FSM dispatches chain mutations as actions; it does not own chain data. Considered `useReducer` only, `useState` + `useReducer`, and a single store for both. Picked the split because the chain is data (CRUD, no FSM-shaped flow) while mode/selection is flow (FSM-shaped, with guards). Both packages share the actor model so integration is one-way (FSM → store).
